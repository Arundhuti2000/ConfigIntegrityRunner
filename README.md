# ConfigIntegrityRunner

A compact PowerShell module and CI pipeline that evaluates an Azure-managed environment against desired-state rules, detects configuration drift, scores integrity, and produces reporting-friendly output for downstream systems including ServiceNow.

---

## Quick Start

```powershell
# Run against sample data (no Azure credentials needed)
./Invoke-IntegrityRunner.ps1 `
    -DesiredStatePath ./sample-data/desired-state.json `
    -ActualStatePath  ./sample-data/actual-state.json  `
    -OutputPath       ./output/report.json             `
    -PassThreshold    80                               `
    -FailOnCriticalDrift
```

**Exit codes:** `0` = gate passed, `1` = score below threshold or critical drift detected.

## Architecture

### Desired State → Actual State → Drift Report

```
desired-state.json          actual-state.json
  (rules + checks)    ───►  (live snapshot)
                              │
                     Compare-ResourceChecks()
                              │
                    per-check diff (recursive)
                              │
                    ┌─────────┴──────────┐
                    │  DriftItems list    │
                    └────────────────────┘
                              │
                    Get-IntegrityScore()
                    (weighted by severity)
                              │
                    ┌────────────────────────┐
                    │  Score, Grade, Summary  │
                    └────────────────────────┘
                              │
               ┌──────────────┴──────────────┐
               │                             │
        integrity-report.json     servicenow-payload.json
        (full detail)              (SNOW Table API ready)
```

### Key Design Decisions

**Recursive comparison engine** — `Compare-ResourceChecks` walks nested objects (like tag blocks) without knowing their schema in advance. Any JSON-shaped desired state works without code changes.

**Severity-weighted scoring** — A Critical NSG drift (open RDP to internet) costs 3× more than a Low-severity cosmetic tag mismatch. This prevents a high volume of low-risk findings from masking a clean-looking score while serious issues go unnoticed.

**Module vs script** — Logic lives in a `.psm1` module with exported functions. The `Invoke-IntegrityRunner.ps1` script is a thin CLI wrapper. This makes the functions independently testable with Pester and reusable from other scripts.

**Separation of desired vs actual state** — Desired state is owned by the platform team in source control. Actual state is a time-stamped snapshot (in production, generated by an Az CLI or Azure PowerShell discovery script). This means the runner can be tested entirely offline.

## Integrity Scoring Model

| Metric               | Formula                                                   |
| -------------------- | --------------------------------------------------------- |
| Per-check weight     | `1 × severity_multiplier`                                 |
| Severity multipliers | Critical=3×, High=2×, Medium=1.5×, Low=1×                 |
| PASS resource        | Earns 100% of weighted points                             |
| DRIFT resource       | Earns `(passing_checks / total_checks)` × weighted points |
| MISSING resource     | Earns 0 points                                            |
| **Score**            | `(earned / total) × 100`, rounded to 1 decimal            |

### Grade Bands

| Grade | Score | Interpretation                                                              |
| ----- | ----- | --------------------------------------------------------------------------- |
| A     | ≥ 95  | Healthy; minor drift acceptable in most contexts                            |
| B     | ≥ 85  | Some drift present; review recommended before next release                  |
| C     | ≥ 70  | Meaningful drift; remediation warranted; escalate if trending down          |
| D     | ≥ 50  | Significant drift; raise ServiceNow incidents; do not promote to production |
| F     | < 50  | Critical failures; immediate action required; consider freeze               |

**Caveat:** Score is a _risk signal_, not a binary pass/fail. A single open-RDP NSG rule (NSG-001 in the sample) should trigger an incident regardless of overall score, which is why `FailOnCriticalDrift` is a separate gate.

## ServiceNow Integration

### Target Tables

| SNOW Table                     | Purpose                                                                | Cardinality |
| ------------------------------ | ---------------------------------------------------------------------- | ----------- |
| `u_azure_config_integrity_run` | One row per pipeline run; stores score, grade, environment, timestamps | 1 per run   |
| `u_azure_config_drift_item`    | One row per drifted check; FK back to run via `u_run_id`               | N per run   |

### How It Flows

1. `ConvertTo-ServiceNowPayload` transforms the run result into two sets of records.
2. A `Push-ToServiceNow.ps1` script (stub in the workflow, ready to implement) POSTs:
   - `POST /api/now/table/u_azure_config_integrity_run` → the run record
   - `POST /api/now/table/u_azure_config_drift_item` (one per drift item) → using the run's `sys_id` as FK
3. SNOW can then:
   - **Dashboard**: Report widget showing score trend over time, pass rate by category
   - **Alerts**: Business rule fires an Incident if `u_score < 50` OR `u_critical_drift > 0`
   - **Change Management**: Link drift records to open Changes; block CAB approval if drift exists in scope
   - **CMDB enrichment**: Drift items can update CI records with a `compliance_status` field

### Recommended SNOW Automation

```
Trigger: After insert on u_azure_config_drift_item
Condition: u_severity = 'Critical'
Action: Create Incident
  - Category: Infrastructure
  - Priority: 2 (High)
  - Short Description: "Azure config drift: {u_resource_name} / {u_check_path}"
  - Description: "Expected: {u_expected_value} | Actual: {u_actual_value}\nRemediation: {u_remediation_hint}"
  - Assignment Group: Platform Engineering
```

## Assumptions

1. **No live Azure environment** — all logic tested against structured JSON files. The schema mirrors what `Az.Compute`/`Az.Network` PowerShell modules return after normalization.
2. **Single subscription scope** — multi-subscription support would require iterating subscriptions and merging actual-state files, which is additive.
3. **Pester 5.x** — the test style uses `BeforeAll`/`BeforeEach` and `Should -Be` syntax from Pester 5, not Pester 4.
4. **ServiceNow custom tables** — custom table names (`u_azure_*`) assume a SNOW instance with the tables already created. In practice, a scoped application would define these.
5. **Static desired state** — for this exercise, desired state is a checked-in JSON file. In production, it would likely be generated from Azure Policy definitions or a Terraform state file.

---

## Major Tradeoffs

| Decision                       | Alternative Considered                       | Why This                                                                   |
| ------------------------------ | -------------------------------------------- | -------------------------------------------------------------------------- |
| JSON desired state             | Azure Policy JSON, Terraform                 | Simpler to version, diff, and review in PRs without AZ toolchain           |
| Flat comparison engine         | Schema-specific validators per resource type | More maintainable; new resource types just need a JSON rule, no code       |
| Weighted score                 | Binary pass/fail                             | More nuanced signal; Critical drift surfaced without drowning in Low noise |
| Pester for tests               | ScriptAnalyzer only                          | Behavior-level tests catch logic bugs that linting cannot                  |
| Separate SNOW payload function | Inline SNOW calls from runner                | Keeps runner offline-testable; SNOW push is optional/conditional           |

## What I Would Do Next

**With more time (priority order):**

1. **`Get-AzureActualState.ps1`** — the discovery script that hits live Azure APIs and produces the actual-state JSON. This is the most important production gap.
2. **`Push-ToServiceNow.ps1`** — the SNOW REST client, including retry logic and idempotent upsert (use `u_run_id` as correlation key).
3. **Drift trending** — compare current run against previous run stored in SNOW; surface "new drift" vs "persisted drift" vs "resolved drift."
4. **Rule inheritance / tagging** — allow rules to be tagged by team/service so reports can be scoped per squad.
5. **HTML report output** — a self-contained HTML summary for attaching to SNOW incidents or emailing stakeholders.
6. **Azure Policy alignment** — map rules to Policy definition IDs so drifts can be correlated with Policy compliance state.
